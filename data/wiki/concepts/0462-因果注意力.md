# 因果注意力

## 定义

因果注意力是一种按序列方向限制信息流的注意力机制：后面的 token 可以读取前面的 token，而前面的 token 不能读取后面的 token。它原本常见于语言模型，但在这些工作中被扩展到视觉、几何重建和自动驾驶任务中，用来把“时间顺序”或“任务顺序”编码进 Transformer 的信息交互方式。

在 OneDrive 中，因果自注意力被用作统一解码器的核心机制：图像 token、检测查询、车道查询、规划查询和文本 token 被拼成一个序列，后续任务 token 通过因果注意力读取前面的视觉信息，从而在同一套预训练 VLM 注意力骨干中完成感知、规划和文本生成 [[paper:24]]。

在 LingBot-Map 中，完整的全历史因果注意力并不是最终选择。作者指出，直接让当前帧关注所有历史帧会带来速度和显存压力，因此用 Geometric Context Attention 将历史拆成 anchor、局部窗口和压缩轨迹记忆，以更结构化的方式保留长程几何上下文 [[paper:8]]。

## 不同视角

一个视角是“序列条件化”。OneDrive 把自动驾驶中的多种输出都组织成一个统一序列，使后面的结构化查询自然以图像 token 为条件。这里的因果注意力主要服务于多任务统一：它让检测、车道、规划、文本生成共享同一套注意力层，并继承预训练 VLM 中已经学到的跨 token 关系建模能力 [[paper:24]]。

另一个视角是“流式记忆控制”。LingBot-Map 面对的是长视频流，如果使用全因果注意力，所有历史帧都会进入注意力计算，成本随历史增长。论文因此把“可看见的过去”分层组织：固定坐标的 anchor、近邻配准窗口、长期轨迹记忆。这里因果性仍然重要，因为当前帧只能利用过去信息，但实现方式从朴素全历史注意力变成了结构化几何上下文 [[paper:8]]。

## 共识与分歧

两篇论文的共识是：因果注意力适合把已有上下文作为条件注入当前预测。无论是自动驾驶中的“图像先于查询”，还是流式重建中的“历史先于当前帧”，核心都是让模型在受控的信息方向下读取上下文，而不是任意双向混合。

分歧在于是否直接使用标准因果注意力。OneDrive 基本保留预训练 LLM/VLM 的因果自注意力，并发现注意力层比 FFN 更容易迁移到驾驶任务 [[paper:24]]。LingBot-Map 则认为在长视频场景中，全因果注意力成本过高，用有界 pose-reference window 替代后，FPS 从 11.87 提升到 20.29，显存从 36.06 GB 降到 13.28 GB，同时 ATE 还从 6.60 降到 5.98 [[paper:8]]。

## 未解问题

这些材料共同暗示了一个开放问题：因果注意力在非文本任务中并没有单一最优形态。对于多任务自动驾驶，直接复用预训练因果注意力有利于统一建模和迁移；对于长时序几何重建，则需要额外设计记忆结构来避免全历史注意力的计算负担。

另一个未完全解决的问题是：哪些信息应该保留为完整 token，哪些应该压缩成记忆。LingBot-Map 通过 anchor、局部窗口和 trajectory memory 给出了一种几何任务中的答案 [[paper:8]]；OneDrive 则通过固定 token 顺序把视觉、结构化查询和文本生成串联起来 [[paper:24]]。但材料中还没有给出一种可跨任务通用的因果注意力组织原则。
