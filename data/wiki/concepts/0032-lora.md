# LoRA

## 定义

LoRA 在这里被归为一种“高效训练主干”的微调技术：其核心角色不是重新训练完整大模型，而是在已有视觉语言模型或多模态主干上进行低成本适配，使模型能够迁移到自动驾驶、3D 空间理解、指代定位、开放世界分割等下游任务。

需要注意的是，给定材料主要描述各论文的任务框架、数据集、空间编码、查询融合、解码器设计和文本监督效果，并未展开 LoRA 的具体矩阵分解形式、插入位置、训练超参或消融结果。因此这里不能补充未给出的 LoRA 技术细节。

## 不同视角

这些论文共同体现了一个背景：大量方法都建立在预训练 VLM/MLLM 之上，再通过少量任务专用模块或适配机制让底座模型服务于新任务。比如 VLM-3R 将几何编码器产生的 3D 场景 token 和相机运动 token 融入视觉语言推理流程 [[paper:3]]；NuGrounding 将 BEV 检测器的 object query 对齐到大语言模型输入空间，用大模型理解自然语言、用检测器提供 3D 几何先验 [[paper:12]]；SpaceDrive 则通过统一 3D 位置编码把视觉 token、文本坐标和轨迹输出连接起来 [[paper:2]]。

从这个角度看，LoRA 所处的位置更像是一类“参数高效适配手段”：它服务于这些以大模型为核心的系统，使模型能在不大规模改动或重训主干的情况下吸收新任务能力。但材料中更明确讨论的是“如何给 VLM 注入空间、几何、驾驶或分割能力”，而不是 LoRA 本身。

## 共识与分歧

材料中的共识是：直接依赖通用 VLM 往往不足以完成精细空间任务，必须通过额外结构或监督增强主干能力。自动驾驶中，SpaceDrive 指出现有 VLM 对三维坐标和轨迹数字生成不可靠，因此引入统一空间编码和回归式轨迹解码 [[paper:2]]；OneDrive 发现预训练 VLM 的注意力层比 FFN 更容易迁移到驾驶任务，并围绕共享注意力构建统一解码器 [[paper:24]]；DriveLM 则通过 Graph VQA 和行为中间层，让驾驶决策从感知、预测、规划逐步推理 [[paper:14]]。

分歧主要不在 LoRA 方案本身，而在“大模型该怎样被适配”。有的方法强调几何 token 融合 [[paper:3]]，有的方法强调 object query 与语言对齐 [[paper:12]]，有的方法强调统一坐标编码 [[paper:2]]，还有方法强调共享解码器和文本监督的正则作用 [[paper:24]]。这些路径都说明：高效微调只是让主干可训练的一部分，真正决定任务表现的还包括输入表示、结构化查询、空间先验和监督形式。

## 未解问题

给定材料没有回答 LoRA 在这些系统中的关键实现问题：例如 LoRA 应插入注意力层、FFN、视觉投影层还是任务适配器；不同任务下 LoRA 与全量微调、adapter、query tuning 的效果差异；以及 LoRA 是否会影响空间推理、3D grounding、轨迹规划或分割边界质量。因此，基于当前材料，只能把 LoRA 概括为“大模型任务适配中的高效微调工具”，不能进一步断言它在这些论文中的具体贡献大小。

## 进一步阅读

若要理解 LoRA 在这些方向中的实际价值，应优先结合论文中的大模型适配部分阅读：关注 VLM-3R 的几何融合 [[paper:3]]、NuGrounding 的 object query 到 MLLM 对齐 [[paper:12]]、SpaceDrive 的空间 token 设计 [[paper:2]]、OneDrive 对注意力迁移能力的诊断 [[paper:24]]，以及 Qwen3-VL-Seg 如何用轻量 decoder 在大模型框先验上生成分割掩码 [[paper:26]]。这些内容能帮助判断：LoRA 适合优化主干迁移成本，但任务能力往往来自更具体的结构设计。
