# SpaceDrive: Infusing Spatial Awareness into VLM-based Autonomous Driving

## 一句话定位

[[SpaceDrive]] 是一项面向[[VLM]]端到端自动驾驶规划的方法，核心是把三维坐标从“文本里的数字”改为统一的[[3D位置编码]]，并同时注入视觉输入、文本提示和轨迹输出过程，从而提升模型的空间感知与规划精度。

## 核心贡献

- 指出当前[[视觉语言模型]]用于自动驾驶时的关键瓶颈：缺乏精细三维空间推理能力，尤其不擅长把坐标准确对应到图像中的目标，也不擅长稳定地产生精确轨迹数字。
- 提出统一的[[3D位置编码]]接口，将空间表示贯穿到感知、提示和输出三端：
  - 图像侧：基于[[深度估计]]获得三维坐标，并将空间编码加到视觉 token 上。
  - 文本侧：将 prompt 中的坐标数字替换为同一种空间编码。
  - 输出侧：用回归解码器直接预测轨迹坐标，而不是逐个数字生成。
- 在 [[nuScenes]] 开环规划和 [[Bench2Drive]] 闭环评测中验证了该设计的有效性；其中开环达到同类最优，闭环表现也很强。

## 方法

### 整体思路

论文的核心思想是：为模型内部建立一套统一的空间坐标表示，而不是继续把坐标当作普通文本数字处理。

一个直观理解是，给每个视觉块附上“带地址的标签”。原本的[[VLM]]更擅长理解“是什么”，但不擅长精确回答“在哪里”；[[SpaceDrive]]则通过统一空间编码，让“语义”和“位置”在同一表示体系中对齐。

### 架构流程

整体流程如下：

1. 输入多路环视图像。
2. [[视觉编码器]]将多视角图像编码为一组 patch 级视觉 token。
3. 另一条支路通过[[深度估计]]预测绝对深度，并将每个 patch 中心反投影为三维坐标。
4. 使用统一的[[3D位置编码]]器将这些坐标编码为空间向量。
5. 将空间向量按元素加到对应视觉 token 上，形成带空间感知的视觉表示。
6. 对文本输入中的坐标，也使用同一套编码进行替换，并添加坐标标记。
7. 推理阶段，普通文本由语言头输出；当需要输出坐标时，隐藏状态送入[[PE解码器]]，直接回归真实三维坐标，用于轨迹规划。

### 关键机制

#### 视觉空间增强

视觉 token 的基本编码形式为：

\[
X_v = f_{vis.}(\{I_k\}) = \{x_p\}_{p=1}^{N}
\]

即先将多视角图像转换为一组 patch 级 token，后续的空间增强都建立在这组 token 上。

随后，对每个 patch 对应的三维坐标 \(c_p\) 做[[三维正弦余弦位置编码]]：

\[
\phi(c_p) = [\phi_x(x_p^{3D}), \phi_y(y_p^{3D}), \phi_z(z_p^{3D})] \in \mathbb{R}^{dim}
\]

其中每个坐标轴都使用正余弦编码，将离散数值变成连续的空间表示，从而保留空间邻近性。

最后，把位置编码注入视觉 token：

\[
\tilde{h}_p = h_p + \alpha_{PE}\,\phi(c_p)
\]

这里的可学习系数 \(\alpha_{PE}\) 用于控制注入强度，避免破坏原始视觉特征分布。这一步对应[[空间token注入]]。

#### 文本坐标替换

论文没有继续将 prompt 中的坐标视为普通数字 token，而是用同一套[[3D位置编码]]把这些坐标直接编码成空间 token。这样做的目的，是让模型在文本侧看到的“坐标”，与视觉侧使用的是同一种空间表示体系。

#### 坐标回归输出

对于轨迹输出，论文不再采用逐位数字生成，而是使用回归式解码：

\[
\hat{c} = \psi(e_{j+1}),\quad \hat{c} \in \mathbb{R}^3
\]

即当模型要输出坐标时，直接从隐藏状态回归三维坐标。这样可以避免语言模型式数字生成带来的误差累积，提高轨迹预测精度与稳定性。

### 相比以往方法的不同

此前相关方法主要有两类：

- 为特定任务设计专门的 3D embedding 或 query；
- 继续沿用语言模型方式，把轨迹点写成数字串并逐 token 生成。

[[SpaceDrive]] 与它们的不同在于：

- 它不是为某一个任务单独加空间模块，而是提供统一的空间表示接口；
- 它不再依赖数字 token 生成坐标，而是采用回归式坐标解码；
- 它同时解决“视觉语义与空间位置对不上”和“数字输出不准”这两个问题。

## 实验与结论

### 开环评测

在 [[nuScenes]] 开环规划中，带 ego 状态的 [[SpaceDrive+]] 在基于[[VLM]]的方法中达到最优：

- 平均 L2 误差：0.32
- 平均碰撞率：0.23%
- 平均越界率：1.27%

对比材料中给出的 [[OmniDrive-Q++]]：

- L2 误差 0.33
- 碰撞率 0.30%
- 越界率 3.00%

这些结果表明，统一[[3D位置编码]]不仅改善轨迹精度，也显著降低了越界风险。

### 闭环评测

在更接近真实驾驶能力的 [[Bench2Drive]] 闭环评测中，[[SpaceDrive+]] 取得：

- Driving Score：78.02
- Success Rate：55.11%

材料指出其在 [[VLM]] 方法中排名第二，表现很强。

### 结论解读

论文强调，一个重要现象是：基础版 [[OmniDrive]] 虽然在开环上表现不错，但闭环成功率不到 10%；而加入显式空间 token 后，性能显著改善。这说明[[SpaceDrive]]带来的提升不只是“更会拟合训练集”，而是增强了模型的实际可驾驶性。

## 限制与待解

- 方法高度依赖前置[[深度估计]]质量；若夜晚、雨雾、反光或遮挡导致深度偏差，后续[[3D位置编码]]也会受到影响。
- 统一[[位置编码]]虽然简洁，但本质上仍是将复杂几何关系压缩进固定维度向量，在极端长距离、密集交互或高度动态场景中，表达能力可能有限。
- 闭环成绩虽强但并非最优，说明仅靠空间编码还不足以完全解决长期时序决策、交互博弈和控制稳定性问题。
- 系统依赖大规模[[VLM]]、深度网络和多阶段接口，训练与部署成本较高，复现对数据处理与坐标对齐也较敏感。
